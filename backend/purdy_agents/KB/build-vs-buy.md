# Build vs. Buy Decision Framework

> **Source**: Derived from PuRDy Instructions

## Overview

This framework helps determine when to build custom solutions vs. purchasing off-the-shelf or hybrid approaches.

---

## Core Questions

1. Is this a unique capability that differentiates my business?
2. What is the true total cost of ownership?
3. Do I have the team to build AND maintain this?
4. What's my exit strategy if this choice doesn't work out?

---

## Decision Matrix

### Build When...

| Factor | Consideration |
|--------|---------------|
| **Competitive Advantage** | The capability IS your product or core differentiator |
| **Unique Requirements** | No existing solution meets critical requirements |
| **Control** | Data sovereignty, security, or compliance requires full control |
| **Long-term Cost** | Build cost + maintenance < subscription over 3-5 years |
| **Team Capability** | You have (or can hire) the skills to build AND maintain |

### Buy When...

| Factor | Consideration |
|--------|---------------|
| **Commodity Function** | The capability is well-solved by existing tools |
| **Speed to Market** | Time-to-value is critical |
| **Maintenance Burden** | You want someone else handling updates, security, uptime |
| **Limited Resources** | Small team, limited budget for ongoing development |
| **Proven Solution** | De-risked, battle-tested by thousands of users |

### Hybrid Approach

| Pattern | When to Use |
|---------|-------------|
| **Buy + Customize** | Core platform + custom integrations/extensions |
| **Build Core, Buy Supporting** | Custom business logic + managed infrastructure |
| **Headless Architecture** | Buy backend, build frontend (or vice versa) |

## Total Cost of Ownership (TCO) Checklist

### Build Costs
- [ ] Initial development time
- [ ] Opportunity cost (what else could the team build?)
- [ ] Infrastructure costs
- [ ] Ongoing maintenance and updates
- [ ] Security patches and compliance
- [ ] On-call and incident response
- [ ] Documentation and training
- [ ] Technical debt accumulation

### Buy Costs
- [ ] Subscription/license fees
- [ ] Implementation and setup
- [ ] Training and onboarding
- [ ] Customization/integration work
- [ ] Vendor lock-in risk
- [ ] Feature gaps requiring workarounds
- [ ] Dependency on vendor roadmap

---

## Red Flags

### Build Red Flags
- "We'll just build it ourselves" without estimating maintenance
- Underestimating integration complexity
- Assuming one developer can "own" it long-term
- Building for hypothetical future requirements

### Buy Red Flags
- Assuming the tool will "just work" for your use case
- Ignoring integration costs
- Not planning for vendor changes/acquisition
- Selecting based on current features, not roadmap alignment
