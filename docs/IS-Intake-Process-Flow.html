🔀 IS Intake Process Flow

Visualizing intake paths and sorting logic for Contentful IS requests

Draft v1.0 - January 2026

1️⃣ First Sort: Project or Support Ticket?

All requests to IS are first sorted into one of two paths based on scope and complexity.

Question If Yes → Project If No → Support
Will this take more than 4 hours of work? ✓
Does it require coordination across multiple teams? ✓
Does it involve multiple systems or data sources? ✓
Does it need executive sponsor approval? ✓
Is it a new capability (not fixing existing)? ✓
Does it require scoping before we know the LOE? ✓
Is it a field edit, permission change, or simple config? ✓

2️⃣ Project Intake Flow (3-Month Cadence)

📥 Project Request Submitted (JIRA)
↓
👔 ELT Acknowledges (Worth scoping?)
↓ YES
📝 BRD Creation
🤖 BuRDy assists here
↓
📊 Scoping & LOE (Month 2)
↓
✅ Executive Approval (Month 3)
↓
🔧 Solution Type Determination

🤖 Where BuRDy Fits

  • Pre-load: Ingest existing docs (Miro, strategy docs, etc.) and pre-fill BRD template
  • Interview: Socratic discovery to fill remaining gaps
  • Output: Completed BRD ready for executive pre-read
  • Time: ~20 minutes per BRD

3️⃣ Solution Type Determination

Once approved, projects are routed to the appropriate solution path.

🤖 Glean Agent

  • Q&A / Search across sources
  • Workflow automation
  • Information synthesis
  • → Agent Factory pipeline

💻 Custom App

  • Custom UI needed
  • Complex business logic
  • Daily-use standalone tool
  • → Custom dev process

🔗 Integration

  • Connect System A to B
  • Data sync between systems
  • API / Workato / Zapier
  • → IS team execution

📋 Process Change

  • Training needed
  • No technology change
  • Behavior/workflow change
  • → Change management

🏢 Vendor Solution

  • Buy not build
  • Mature market solution
  • Time-to-value critical
  • → RFP / POC process

4️⃣ Is This a Glean Agent?

Question If Yes → Likely Glean Agent
Is it primarily about answering questions from company knowledge? ✓
Does it involve searching across multiple data sources? ✓
Is it automating a research/lookup workflow? ✓
Does it synthesize information for users? ✓
Can it be triggered by chat, form, or schedule? ✓
Does it NOT require real-time data writes to external systems? ✓

Example Glean Agent Use Cases:

  • "Give me a 360 view of this customer account"
  • "What's our policy on X?"
  • "Summarize the status of project Y"
  • "Find experts on topic Z"

❓ Open Questions for Further Design

  1. Triage Ownership: Who does the initial project vs. support sort? IS PMO? Individual IS teams?
  2. Solution Type Determination: When is solution type determined?
    • Option A: During BRD creation (BuRDy asks and routes)
    • Option B: After BRD, during scoping
    • Option C: IS analyst makes the call based on BRD content
  3. Hybrid Solutions: What if a project needs multiple solution types? (e.g., Glean Agent + Integration)
  4. Fast-Track Path: Is there a fast-track for urgent/compliance/revenue-impacting work?
  5. Pre-BRD Triage: Should there be a lighter-weight triage before full BRD to determine solution type?
  6. Agent Factory Entry: Can users go directly to Agent Factory if they already know they want a Glean Agent?
