- Simple, geometric, two-tone

## Stage 0: Starting Inputs

### Detailed Information

**Purpose:** Gather the raw materials that frame the problem space before any DISCo work begins.

**Key Inputs:**

| Input | Description | Why It Matters |
|-------|-------------|----------------|
| Problem Statement | Initial hypothesis of what needs solving | Focuses triage and discovery planning |
| Existing Research | Market research, competitive analysis, prior studies | Avoids rediscovering known information |
| Golden Examples | Exemplary solutions or analogous successes | Provides aspirational targets and patterns |
| Stakeholder Map | Who's affected, who decides, who implements | Ensures right people are interviewed |
| Constraints | Budget, timeline, technical limits, regulations | Bounds the solution space early |
| Supporting Documents | Emails, tickets, complaints, feature requests | Raw evidence of the problem |

**Quality Checklist:**
- [ ] Problem statement is specific, not vague
- [ ] Research is recent and relevant
- [ ] Golden examples are truly analogous
- [ ] Stakeholder map includes skeptics, not just champions
- [ ] Constraints are confirmed, not assumed
- [ ] Supporting documents represent multiple perspectives

**Output:** A populated DISCo initiative ready for Triage

## Stage 1: Discovery (Intake & Validation)

### Detailed Information

**Purpose:** Validate whether the problem is worth solving and plan how to investigate it through human-led sessions.

**Two Agents:**

| Agent | Role | Key Question | Output |
|-------|------|--------------|--------|
| Triage | GO/NO-GO decision | Is this problem worth solving? | Decision + routing recommendation |
| Discovery Planner | Session planning | What do we need to learn? | Interview guides, workshop agendas |

**Triage: Problem Worth Solving Gate**

Four criteria (all must be YES for GO):
1. **Real**: Does this problem actually exist? Evidence?
2. **Costly**: Is it painful enough to warrant solving?
3. **Solvable**: Can we realistically address it?
4. **Ours**: Are we the right team to solve it?

Possible decisions:
- **GO**: Proceed to Discovery Planner
- **NO-GO**: Document why, archive initiative
- **INVESTIGATE**: Need more input before deciding

**Discovery Planner Outputs:**
- Session agenda with time allocations
- Interview questions by stakeholder type
- Workshop frameworks (journey mapping, affinity diagrams, etc.)
- Document request list
- Recommended session sequence

**Human Discovery Sessions:**

| Session Type | Purpose | Typical Duration |
|--------------|---------|------------------|
| Stakeholder Interview | Deep dive with individual | 45-60 min |
| Group Workshop | Collaborative exploration | 2-4 hours |
| Observation Session | Watch users in context | 1-2 hours |
| Document Review | Analyze existing materials | Async |

**What Humans Capture:**
- Direct quotes and stories
- Emotional reactions and body language
- Workarounds and unofficial processes
- Contradictions between stated and observed
- Political dynamics and relationships

**Output:** Session recordings, notes, artifacts uploaded to DISCo

```

## Stage 2: Insights (Analysis & Consolidation)

### Detailed Information

**Purpose:** Extract meaning from discovery sessions—find patterns, identify gaps, and synthesize findings into a decision document.

**Three Agents:**

| Agent | Role | Key Question | Output |
|-------|------|--------------|--------|
| Coverage Tracker | Gap analysis | Have we learned enough? | Readiness status, gap list |
| Insight Extractor | Pattern recognition | What patterns emerged? | Pattern library, contradictions |
| Consolidator | Decision synthesis | What's the recommendation? | Decision document with metrics |

**Coverage Tracker:**

Assesses completeness across dimensions:
- Stakeholder coverage (have we talked to all key groups?)
- Topic coverage (have we explored all aspects?)
- Evidence quality (do we have sufficient depth?)

Status outputs:
- **READY FOR SYNTHESIS**: Sufficient coverage, proceed
- **GAPS REMAIN (CRITICAL)**: Missing essential perspectives
- **GAPS REMAIN (MINOR)**: Could proceed, but recommend more
- **BLOCKED**: Cannot proceed until specific blocker resolved

**Insight Extractor:**

Identifies five types of enterprise dynamics:
1. **Reinforcing Loops**: Behaviors that amplify themselves
2. **Balancing Loops**: Self-correcting patterns
3. **Delays**: Time gaps between action and effect
4. **Workarounds**: Unofficial solutions revealing unmet needs
5. **Contradictions**: Conflicting information requiring resolution

Output includes:
- Pattern library with named patterns
- "What they don't realize" insights
- Information quality assessment
- Synthesis recommendations

**Consolidator:**

Creates comprehensive decision document (900 words max):
- **Decision**: First word is GO/NO-GO/CONDITIONAL
- **Leverage Point**: Where intervention will have most impact
- **System Diagram**: Visual of dynamics with intervention reasoning
- **Metrics Dashboard**: How we'll measure success
- **Evidence Summary**: Key supporting findings
- **Blockers**: What could prevent success
- **Action Plan**: Recommended next steps

**Output:** Reviewed decision document ready for Synthesis phase

```

CONTENT TO SHOW:
- Three-agent pipeline: Coverage Tracker → Insight Extractor → Consolidator

## Stage 3: Synthesis (Bundling & Initiative Definition)

### Detailed Information

**Purpose:** Cluster related insights into implementation bundles that humans can approve, reject, merge, or split.

**One Agent:**

| Agent | Role | Key Question | Output |
|-------|------|--------------|--------|
| Strategist | Initiative bundling | What should we build? | Scored bundles with dependencies |

**Clustering Logic:**

The Strategist groups findings by:
- **Theme**: Related problem areas
- **Root Cause**: Issues with shared underlying cause
- **Solution Affinity**: Problems addressable by same solution
- **Stakeholder Impact**: Issues affecting same groups

**Bundle Scoring (IFU Framework):**

| Dimension | Factors | Score Range |
|-----------|---------|-------------|
| **Impact** | People affected, workflow criticality, financial impact | 1-5 |
| **Feasibility** | Path clarity, data availability, technical complexity | 1-5 |
| **Urgency** | Regulatory pressure, competitive pressure, stakeholder demand | 1-5 |

Each score includes rationale explaining the assessment.

**Bundle Structure:**

Each proposed bundle contains:
- **Name**: Descriptive title
- **Description**: What this initiative addresses
- **Included Items**: Which insights/findings are grouped here
- **Affected Stakeholders**: Who benefits, who's impacted
- **Complexity Tier**: Light / Medium / Heavy
- **Dependencies**: Blocks, requires, conflicts with other bundles

**Human Checkpoint Actions:**

| Action | Description |
|--------|-------------|
| **Approve** | Bundle proceeds to PRD generation |
| **Reject** | Bundle archived with feedback |
| **Merge** | Combine two bundles into one |
| **Split** | Divide bundle into smaller initiatives |
| **Adjust Score** | Override AI scoring with rationale |
| **Add/Remove Items** | Modify what's included in bundle |

**Output:** Approved bundles ready for PRD generation

## Stage 4: Convergence (PRD Generation)

### Detailed Information

**Purpose:** Transform approved bundles into engineering-ready Product Requirements Documents with technical guidance.

**Two Agents:**

| Agent | Role | Key Question | Output |
|-------|------|--------------|--------|
| PRD Generator | Specification writing | What exactly should we build? | Engineering-ready PRD |
| Tech Evaluation | Architecture guidance | How should we build it? | Build vs. buy analysis |

**PRD Generator Output Structure:**

| Section | Content |
|---------|---------|
| Executive Summary | One-paragraph overview for leadership |
| Problem Statement | What we're solving and why |
| Root Causes | Underlying issues from discovery |
| Success Criteria | Measurable outcomes (SMART format) |
| User Stories | Who does what to achieve which goal |
| Feature Requirements | Detailed functional specifications |
| Acceptance Criteria | How we know each feature is complete |
| Dependencies | What must exist or happen first |
| Roadmap | Phased delivery approach |
| Rollback Strategy | How to reverse if needed |

**Tech Evaluation Output:**

| Section | Content |
|---------|---------|
| Build vs. Buy Analysis | Options with trade-offs |
| Recommended Approach | Clear recommendation with rationale |
| Architecture Diagram | System components and interactions |
| Integration Points | How it connects to existing systems |
| Technology Stack | Recommended technologies |
| Risk Assessment | Technical risks and mitigations |
| Resource Estimate | Team composition and skills needed |

**PRD Workflow:**

1. PRD Generator creates draft from approved bundle
2. Human reviews and edits PRD
3. Optionally: Tech Evaluation adds technical guidance
4. Human approves final PRD
5. PRD exported for engineering handoff

**Complexity Tiers Guide Depth:**

| Tier | PRD Depth | Tech Eval Depth |
|------|-----------|-----------------|
| Light | Concise, 2-3 pages | Optional, high-level |
| Medium | Standard, 5-8 pages | Recommended, moderate detail |
| Heavy | Comprehensive, 10+ pages | Required, full architecture |

**Output:** Approved PRDs ready for engineering

## Stage 5: Operationalization

### Detailed Information

**Purpose:** Take approved PRDs into implementation, deployment, and measurement—closing the loop from discovery to delivered value.

**Key Activities:**

| Phase | Activities | Outputs |
|-------|------------|---------|
| Implementation | Sprint planning, development, code review, testing | Working software |
| Deployment | Staged rollout (pilot → limited → full), monitoring | Production release |
| Measurement | Track success criteria, collect feedback, analyze outcomes | Performance data |
| Iteration | Adjust based on learnings, close or continue | Lessons learned |

**Implementation Workflow:**

1. PRD handed off to engineering team
2. Work broken into sprints/iterations
3. Integration with existing systems validated
4. QA and acceptance testing against PRD criteria
5. Stakeholder demo and sign-off

**Deployment Strategy:**

| Stage | Scope | Purpose |
|-------|-------|---------|
| Pilot | Single team or small group | Validate in real conditions |
| Limited | Expanded audience | Test at scale, gather feedback |
| Full | All intended users | Complete rollout |

Each stage includes go/no-go decision before proceeding.

**Measurement Framework:**

Track against success criteria defined in PRD:
- **Adoption metrics**: Usage, engagement, retention
- **Outcome metrics**: The business results we aimed to achieve
- **Feedback signals**: Support tickets, surveys, user interviews
- **Technical health**: Performance, errors, reliability

**Iteration Decisions:**

| Signal | Response |
|--------|----------|
| Metrics on track | Continue monitoring, eventual close |
| Minor gaps | Iterate within current initiative |
| Major gaps | New DISCo discovery cycle |
| Unexpected success | Consider expansion initiative |

**Output:** Working solution in production with measured outcomes and captured lessons

## Complete Series Overview

When creating all six infographics, ensure:

1. **Consistent header treatment** with stage number and name
2. **Visual continuity** in how outputs from one stage become inputs to the next
3. **Human vs. AI distinction** clear throughout (humans in coral, AI in navy)
4. **Checkpoint markers** at each human decision point
5. **Progress indicator** showing where this stage fits in overall flow

Suggested series layout:
```
[Stage 0: Inputs] → [Stage 1: Discovery] → [Stage 2: Insights] → [Stage 3: Synthesis] → [Stage 4: Convergence] → [Stage 5: Operationalization]
     Gather           Human Sessions         AI Analysis          Human Decisions        Engineering PRDs           Deliver & Measure
```
