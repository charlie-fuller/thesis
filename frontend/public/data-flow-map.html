Layer 2: Document Processing Pipeline

Every document goes through a multi-step processing pipeline before becoming searchable and usable by agents.

Processing Steps

Raw document → Text extraction → Chunking (1000 chars) → Auto-classification → Voyage AI embeddings → Store vectors

Text Extraction

  • PDF: PyPDF2 page-by-page extraction
  • Markdown: Frontmatter parsed, body cleaned
  • TXT: Raw text with encoding detection
  • Metadata preserved (title, author, dates)

Auto-Classification

  • Claude classifies document type automatically
  • Types: transcript, notes, report, analysis, planning, reference
  • Tags extracted from content and frontmatter
  • Folder path preserved for organization

Embedding Generation

  • Voyage AI voyage-3 model (1024 dimensions)
  • Chunks: 1000 chars with 200 char overlap
  • Title + heading prepended for context
  • Stored as pgvector in Supabase

Knowledge Graph

  • Neo4j Aura for entity relationships
  • People, organizations, topics as nodes
  • Documents linked to entities they mention
  • Used by agents for relationship queries

Layer 3: Storage Layer

Two complementary databases serve different query patterns: Supabase for structured data and vector search, Neo4j for relationship traversal.

Supabase (PostgreSQL)

  • kb_documents: Full documents with metadata
  • kb_document_chunks: Embedded chunks for vector search
  • project_tasks: Tasks with status, assignee, priority
  • projects: Scored opportunities with tiers
  • stakeholders: People with engagement tracking
  • disco_initiatives: DISCO investigations
  • conversations / messages: Chat history
  • help_documents / help_chunks: RAG help system

Neo4j Aura (Graph DB)

  • Person nodes: stakeholders, attendees
  • Organization nodes: companies, teams
  • Topic nodes: themes, technologies
  • Document nodes: linked to KB docs
  • MENTIONS: document mentions entity
  • WORKS_WITH: person-organization links
  • Used for: "Who knows about X?", "What's related to Y?"
Vector Search Pattern: When an agent needs context, it performs a cosine similarity search over kb_document_chunks using Voyage AI embeddings. The top-k chunks (usually 5-10) are injected into the agent's prompt as grounding context. This is why embedding quality matters.

Layer 4: Feature Modules

Six major features consume the stored data in different ways. Each module reads from specific tables and writes back structured results.

DISCO Initiatives

  • Links KB docs to initiatives
  • 4-stage pipeline (D-I-S-C)
  • Throughline framing guides analysis
  • Outputs: insights, requirements, projects

Chat + Meeting Rooms

  • 22 agents with KB vector context
  • @mention routing to specialists
  • Meeting rooms: multi-agent discussion
  • Conversation history persisted

Projects Pipeline

  • 4-dimension scoring (ROI, Effort, etc.)
  • Tier auto-calculation (T1-T4)
  • Status lifecycle tracking
  • Links to tasks and documents

Task Management

  • Kanban board (To Do thru Done)
  • Created from DISCO, Taskmaster, manual
  • Linked to projects and stakeholders
  • Priority, assignee, due date tracking

Pipeline Aggregation

  • Aggregates projects by status/tier
  • Team workload visualization
  • Status distribution analytics
  • Filtering by team, tier, status

Intelligence Module

  • Stakeholder engagement scoring
  • Relationship mapping via Neo4j
  • Engagement history tracking
  • Champion / blocker identification

Layer 5: AI Agent Layer

22 specialized agents process requests using KB context, conversation history, and structured data. Each agent has a specific domain and XML-defined behavior.

How Agents Access Data

User message → Agent router → Vector search KB → Graph query → Claude + context → Response

Agent Categories

  • Stakeholder: CTO, CFO, CISO, CDO perspectives
  • Consulting: Oracle, Strategist, Architect
  • Enablement: Adoption, Change, Training
  • Task Mgmt: Taskmaster, Kraken
  • Systems: Facilitator, Reporter, Librarian
  • Personal: Career, Manual, Throughline

Context Injection

  • Vector search: top-k chunks by cosine similarity
  • Graph context: related entities from Neo4j
  • Conversation history: last N messages
  • Structured data: relevant tasks, projects, stakeholders
  • System instructions: XML prompt with RCCI framework
  • Smart brevity: 100-150 word response target

Agent Actions

  • Respond to chat messages with analysis
  • Create tasks from conversation context
  • Generate DISCO stage outputs
  • Score projects and evaluate with Kraken
  • Extract stakeholders from documents
  • Synthesize meeting room discussions

Cross-Cutting Data Connections

Features don't operate in isolation. Here's how data flows between modules, creating a connected intelligence network.

DISCO to Projects

DISCO Discovery → Insights generated → Requirements clustered → Projects proposed → Pipeline tracked

Meeting Notes to Tasks

Granola note synced → Action items extracted → Tasks created → Stakeholders linked

KB to Intelligence

KB documents → Entity extraction → Neo4j graph → Relationship map → Engagement scores
Feedback Loop: Agent outputs (analyses, recommendations, summaries) can be saved back to the KB as documents, creating a growing institutional knowledge base that improves future agent responses.

Key Principles

KB is the Foundation

Everything starts with documents. The richer the KB, the better agents perform. Vault sync keeps it current.

Everything Links

Documents link to initiatives, initiatives produce projects, projects contain tasks, tasks reference stakeholders. The graph grows naturally.

Agents Use Context

Every agent response is grounded in your KB documents via vector search. They cite sources and prioritize your data over general knowledge.

Data Flows Down

Strategic analysis (DISCO) produces tactical output (projects, tasks). Top-down: initiative framing guides discovery, which guides project creation.

Two Storage Models

Supabase for structured records and vector search. Neo4j for relationship traversal. Both queried by agents as needed.

Continuous Enrichment

Every sync, upload, meeting, and agent interaction adds to the knowledge base. The platform gets smarter as you use it.
