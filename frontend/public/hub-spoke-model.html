Hub-and-Spoke Governance Model
Teams can use preferred tools within standards -- the hub provides governance, the spokes provide flexibility
Hub: Central Standards
Standard Configuration Files

Every tool/agent must have: standard MD documentation, JSON config where applicable, KB entry in Thesis platform, evaluation criteria defined

Accountability Requirements

Process lead identified, backup owner named, evaluation cadence set, handoff documentation maintained

Security Baseline

Enterprise agreement with vendor, proper authentication (not just password), no uncontrolled external sharing, data classification applied

Spokes: Team Flexibility
Approved Platform Set

Glean, Gemini/Gems, NotebookLM, Claude (with approval), MuleSoft (enterprise), Custom (with justification)

Experimentation Path

No-cost + less than 1 month: experimentation ticket. Requires budget: procurement process. New vendor: security + legal review.

Evaluation Before Deployment

Test against defined criteria before release. Ongoing evaluation cadence after deployment. Standard QA process for agents and workflows.

MCP Architecture Note (from Raza): MCPs should be designed per use case, not as 1:1 API replacements. "If I say 'give me all pull requests during November 2025,' the MCP should know exactly which tool to use. Don't expose raw API endpoints as MCP tools -- that makes things less deterministic, not more."
